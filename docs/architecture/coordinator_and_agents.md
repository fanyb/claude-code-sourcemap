# 多 Agent 协作架构 (Coordinator & Agents)

Claude Code 支持一种高级的“协调器模式”(Coordinator Mode)，通过 `AgentTool` 实现任务的并行拆解和多 Agent 协同。

## 核心概念

### 1. Coordinator (协调器)
- **角色**：作为主 Agent，负责接收用户的高层指令（例如：“重构整个项目的日志系统”）。
- **职责**：规划任务、通过 `AgentTool` 派生子 Agent、汇总子 Agent 的执行结果。
- **限制**：Coordinator 通常被限制直接使用 `Bash` 或 `Edit` 工具，它必须通过子 Agent 来操作文件系统，这增加了操作的安全性与规划的严谨性。

### 2. Sub-Agent (子 Agent)
- **角色**：执行特定任务的原子单元。
- **独立性**：每个子 Agent 拥有独立的：
  - 消息历史 (Conversation History)
  - 工作目录上下文
  - 独立的工具调用权限
- **通信**：子 Agent 执行完毕后，将结果（文本摘要或状态）返回给父 Agent。

## 协作流程图

```text
[用户指令: 重构 A 和 B 模块]
       |
       v
+--------------+
| Coordinator  | (规划中...)
+------+-------+
       |
       +-----------------------+-----------------------+
       |                       |                       |
+------v-------+        +------v-------+        +------v-------+
| Sub-Agent A  |        | Sub-Agent B  |        | Sub-Agent C  |
| (修改 A 文件) |        | (修改 B 文件) |        | (运行测试)   |
+------+-------+        +------+-------+        +------v-------+
       |                       |                       |
       +-----------+-----------+-----------------------+
                   |
            +------v-------+
            | Coordinator  | (汇总结果，回复用户)
            +--------------+
```

## 关键技术点

### AgentTool
- **派生机制**：模型调用 `Agent(name, goal)` 时，系统会启动一个新的 `QueryEngine` 实例。
- **并发控制**：主 Agent 可以根据配置同时运行多个子 Agent。
- **上下文继承**：子 Agent 会继承父 Agent 的部分环境配置（如权限模式），但不会共享完整的消息历史，以防止上下文爆炸。

### 状态隔离
- 所有的子 Agent 任务都被记录在 `state/AppStateStore.ts` 中，UI 会展示一个实时的任务树状图，用户可以监控每个子 Agent 的进度。
- 子 Agent 失败不会导致主 Agent 崩溃，主 Agent 可以尝试重试或指派另一个 Agent 处理。
