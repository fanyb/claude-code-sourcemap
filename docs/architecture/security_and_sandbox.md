# 沙箱与安全隔离 (Security & Sandbox)

终端 AI 助手拥有操作宿主文件系统和执行 Shell 命令的能力。为了防止恶意 Prompt 注入或操作失误导致数据损坏，Claude Code 设计了严密的沙箱与安全隔离机制。

## 1. 权限模式 (Permission Modes)

核心权限逻辑位于 `src/utils/permissions/`。

### 模式分类
- **Prompt (每次询问)**: 任何非只读的工具调用前，必须弹出 UI 请求用户批准。
- **Auto (自动)**: 自动放行绝大部分工具，但如果触发了高危指令特征库（Classifier），则降级为 Prompt。
- **Bypass (完全信任)**: 在特定的非交互式环境（如 CI/CD）中使用，跳过所有检查。

## 2. 危险命令分类器 (Danger Classifier)

当处于 Auto 模式时，如何判断一个 Bash 命令是安全的？
- **特征匹配**：系统内部维护了一个轻量级的规则引擎，用于拦截如 `rm -rf /`, `mkfs`, 或尝试读取 SSH 密钥、钱包凭证的敏感操作。
- 当分类器识别到风险时，会强行打断自动执行流程，并在 UI 中标红提示风险级别。

## 3. 容器沙箱 (Docker Sandbox Adapter)

位于 `src/utils/sandbox/`，这是为最高安全级别设计的隔离层：
- **原理**：不直接在宿主机（Host）上执行 `run_shell_command`，而是通过内部适配器将命令路由到一个隔离的 Docker 容器或轻量级 VM 中执行。
- **网络隔离**：可以通过工具层面的拦截，阻断模型发起的未知网络请求，防止敏感代码被外泄（Data Exfiltration）。
- **同步问题**：需要在沙箱内外进行双向的文件状态同步（例如：AI 在沙箱内修改了文件，需要将修改映射回宿主机的项目目录）。

## 4. 会话恢复安全
系统在从崩溃中恢复历史快照时，会自动清空历史中的“待审批”工具调用，防止因为界面错乱导致用户意外批准了被截断的危险命令。
